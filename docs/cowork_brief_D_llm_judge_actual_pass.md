# Cowork Brief D: LLM judge actual pass

## One-sentence task
Run `scripts/grade_traces.py` (already shipped via PR #83) against 30-50 real traces covering T1 decks and produce `results/llm_audit_report.md` with systematic strategy weaknesses ranked by severity.

## Why this matters
The LLM judge infrastructure exists but has only been run against 2 sample traces (storm_vs_dnt seeds 42 + 99). Running it at scale reveals:
- Which strategies consistently grade poorly in which domain (e.g. "storm.mana consistently C under Thalia")
- Which matchups the sim plays badly regardless of deck (LLM grader may flag systemic issues)
- Calibration targets for future fixes

This is essentially a no-code data-collection PR that turns the existing tool into a deliverable audit.

## Scope

### Part 1 — Trace collection (N=30-50 games)

Use `llm_judge.py collect` for a representative sample. Suggested coverage:

```bash
# 3 seeds × 10 matchups = 30 traces
# Matchups span the meta (T1, T2, and both combo and fair)
for m in "storm dnt" "oops burn" "burn storm" "dimir ur_delver" \
         "reanimator dimir" "eldrazi storm" "prison dimir" \
         "lands burn" "ur_delver affinity" "dnt storm"; do
  python3 llm_judge.py collect $m --seeds 42 99 2026
done
```

Ensure each trace file ends up in `results/traces/*.json`.

### Part 2 — Grade

```bash
export ANTHROPIC_API_KEY=sk-ant-...
python3 scripts/grade_traces.py results/traces/*.json
# produces results/traces/*_graded.json per trace (~30 files)
python3 scripts/grade_traces.py --report
# produces results/llm_audit_report.md
```

Cost check: at ~4KB per prompt × 30 traces × Claude input rate ≈ < $1 total.

### Part 3 — Analysis

The `--report` output already produces:
- Domain averages across all traces
- Worst/best per domain
- Per-trace summaries
- Flagged weaknesses (3+ C/D grades in same `<deck>.<domain>`)

Write a short companion doc `results/llm_audit_findings_<date>.md` that:
- Summarizes the top 5 flagged weaknesses
- For each, proposes a concrete fix (code change or gameplan tweak)
- Sorts by expected WR impact

### Part 4 — CI integration (optional)

Add a GitHub Action that runs `scripts/grade_traces.py --report --threshold B-` on a scheduled basis (weekly) against a fixed trace set committed to the repo. If the average grade drops below B- in any domain, fail the job — we regressed AI quality.

Only do Part 4 if time permits.

## Constraints
- Don't commit the API key. `export` it in your shell.
- Cap at ~50 traces total to keep API cost trivial.
- Don't commit every prompt file — only the `_graded.json` files + the markdown report. Prompt files are derivable from traces.
